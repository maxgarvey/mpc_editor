# internal/db

SQLite persistence layer for user preferences, workspace file catalog, and file metadata.

## Purpose

Stores everything that needs to survive a restart: user preferences, the indexed catalog of files in the workspace (WAV/PGM/SEQ), per-file metadata (BPM, sample rate, pad-sample assignments), and file tags. The schema is managed in-process — no external migration tool.

## Files

| File | Role |
|------|------|
| `db.go` | `Open()` — opens/creates the SQLite DB, runs schema DDL, runs migrations |
| `schema.sql` | Canonical schema DDL (embedded via `//go:embed`) |
| `queries.sql` | Named sqlc queries — source of truth for all SQL |
| `queries.sql.go` | Generated by `sqlc` from `queries.sql` — do not edit by hand |
| `models.go` | Generated row types |
| `migrate.go` | Additive `ALTER TABLE` / `CREATE TABLE IF NOT EXISTS` migration functions run at startup |

## Database Location

`~/.mpc_editor/mpc_editor.db` — created on first run.

## Schema Overview

```
preferences      — single-row user settings (profile, workspace path, last opened paths)
files            — workspace file catalog (path, type, size, mod_time)
wav_meta         — WAV metadata per file (sample_rate, channels, bits, frame_count, source URL)
pgm_meta         — PGM metadata (midi_pgm_change)
pgm_samples      — pad×layer sample name assignments extracted from each .pgm
seq_meta         — SEQ metadata (bpm, bars, version)
seq_tracks       — track metadata extracted from each .seq (name, MIDI channel, pgm reference)
song_steps       — .sng song step list (step, seq reference, repeats, tempo)
file_tags        — arbitrary key/value tags per file (auto-generated or user-set)
```

## Connection Pool

The pool is capped at **one open connection** (`SetMaxOpenConns(1)`). SQLite supports only one concurrent writer; a pool of one lets Go's `database/sql` serialize all operations, preventing `SQLITE_BUSY` errors between the background workspace scanner and UI writes.

## Migration Strategy

Migrations are inline Go functions (`migrateAdd*`, `migrateCreate*`) called in `Open()` after the base schema DDL. Each migration uses `ALTER TABLE ... ADD COLUMN` or `CREATE TABLE IF NOT EXISTS` — both are idempotent and safe to run on every startup. Errors are ignored (they mean the column/table already exists).

A one-time data migration (`migrateJSONPrefs`) reads the old `preferences.json` file, writes it to the DB, and renames the JSON file to `.bak`.

## Regenerating Queries

After editing `queries.sql` or `schema.sql`:

```bash
make generate   # runs sqlc generate
```

Requires `sqlc` to be installed. The generated `queries.sql.go` and `models.go` should be committed.
